API Documentation

Barangay Management System API v1

This documentation covers the REST API used by the PickerKarma Barangay Management System. The API provides token-based authentication, resident-facing services, and admin-only management endpoints.

Audience: frontend developers, mobile app developers, QA testers, and system integrators.

Base URL

https://your-domain.com/api/v1

Local development example: http://127.0.0.1:8000/api/v1

Authentication Mechanism

The API uses Laravel Sanctum Personal Access Tokens with Bearer authentication. Log in through /auth/login to receive a token, then send it in the Authorization header. 

Authorization: Bearer {token}
  • auth:sanctum protects authenticated routes.
  • resident.verified ensures resident accounts are fully verified.
  • admin middleware limits admin modules to admin users only.

Request and Response Format

Standard Request Headers

Accept: application/json
Content-Type: application/json
Authorization: Bearer {token}

Request Body Example

{
  "title": "Barangay Clean-up Drive",
  "description": "Community event for all residents.",
  "starts_at": "2026-05-15T08:00:00+08:00",
  "ends_at": "2026-05-15T12:00:00+08:00",
  "is_paid": true,
  "registration_fee": 100
}

Success Response Example

{
  "message": "Event created successfully.",
  "event": {
    "id": 12,
    "title": "Barangay Clean-up Drive",
    "is_live_now": false
  }
}

Error Codes

Status Meaning Typical Cause
200 / 201 Success Request completed successfully.
401 Unauthorized Missing or invalid token.
403 Forbidden User lacks required role or verification status.
404 Not Found Requested record does not exist.
419 Session/CSRF issue Mostly seen in web-form contexts.
422 Validation Error Invalid request parameters or payload.
500 Server Error Unhandled backend exception.

Auth Endpoints

POST /auth/login

Authenticate user and create Sanctum token.

Request Body

{
  "email": "resident@example.com",
  "password": "your-password",
  "device_name": "web-client"
}

Response

{
  "token": "1|token-value",
  "token_type": "Bearer",
  "user": { "id": 3, "name": "Juan Dela Cruz", "role": "resident" }
}
GET /auth/me

Get currently authenticated user profile.

POST /auth/logout

Revoke current access token.

Resident Event Endpoints

GET /events

List discoverable community events for verified residents.

Query Parameters

  • status (optional): upcoming, live, completed
  • search (optional): keyword search by event title
GET /events/{event}

Get full event details including schedule, fee, and registration status context.

POST /events/{event}/register

Register current resident to an event. Returns direct registration for free events or Stripe checkout URL for paid events.

Response Notes

  • Free events: returns completed registration payload.
  • Paid events: returns checkout_url for payment.

Resident Incident Endpoints

GET /incidents

List incidents for current resident.

Query Parameters

  • scope (optional): filed or respondent
POST /incidents

Submit a new incident report.

Request Body

{
  "category": "Noise Complaint",
  "description": "Loud music past midnight.",
  "incident_date": "2026-04-25",
  "respondent_email": "neighbor@example.com"
}
GET /incidents/{incident}

Get a single incident report if owned by current resident context.

Resident Document Endpoints

GET /documents

List resident document requests and statuses.

POST /documents

Create document request and return Stripe checkout URL when payment is required.

Request Body

{
  "document_type": "Barangay Clearance",
  "purpose": "Employment requirement",
  "quantity": 1
}
GET /documents/{documentRequest}

Get details of one document request.

Notification Endpoints

GET /notifications

List authenticated user notifications.

POST /notifications/read-all

Mark all current user notifications as read.

POST /notifications/{notification}/read

Mark one notification as read.

Admin Event Endpoints

All routes in this section require admin role + Sanctum token.

GET /admin/community-events

List all community events for administration.

POST /admin/community-events

Create event and trigger notification workflow.

GET /admin/community-events/{event}

View event details and management metadata.

PUT /admin/community-events/{event}

Update event details.

DELETE /admin/community-events/{event}

Delete event record.

POST /admin/community-events/{event}/live

Toggle NOW LIVE status used by QR check-in validation.

POST /admin/community-events/{event}/simulate-registrations

Generate sample registrations for testing and demonstrations.

GET /admin/community-events/{event}/registrations

List event registrants including payment and check-in status.

Admin Incident Endpoints

GET /admin/incidents

List all filed incidents for review.

GET /admin/incidents/{incident}

View complete incident details.

POST /admin/incidents/{incident}/review

Accept or reject incident with outcome and notes.

Admin Document Endpoints

GET /admin/documents

List all document requests submitted by residents.

GET /admin/documents/{documentRequest}

Get one document request details.

POST /admin/documents/{documentRequest}/approve

Approve document request and move to fulfillment.

POST /admin/documents/{documentRequest}/reject

Reject document request and handle refund workflow when applicable.

OpenAPI Specification Endpoint

GET /docs/openapi.yaml

Returns OpenAPI 3.0.3 YAML definition of available API routes.

Quick cURL Example

curl -H "Accept: application/json" https://your-domain.com/api/v1/events

Testing and Demonstration

This is the exact test flow we used while developing the API. It is written as an execution log so panel members can see what was called, what should return, and what output proves the integration is working.

Pre-check Before Running API Tests

  • Backend server running (Laravel via XAMPP/Apache or php artisan serve).
  • Public URL active (ngrok) if testing outside local machine.
  • Seeded users available (AdminAndResidentsSeeder) for resident/admin credentials.
  • Accept: application/json is always included in request headers.

1) API Requests and Responses

Start from unauthenticated route, then move to protected resident routes, then admin routes.

Step 1 - Login and Capture Token

POST /api/v1/auth/login

Expected: 200 OK
Expected keys: token, token_type, user
{
  "token": "1|sanctum-token",
  "token_type": "Bearer",
  "user": {
    "id": 7,
    "name": "Resident 07",
    "email": "resident1@example.com",
    "role": "resident"
  }
}

Step 2 - Verify Protected Route Works With Token

GET /api/v1/auth/me

Expected: 200 OK
Expected: same user id/email returned from login token owner

Step 3 - Verify Resident Business Endpoint

GET /api/v1/events

Expected: 200 OK
Expected: list payload (can be empty array if no events yet)

Step 4 - Verify Middleware Rejects Invalid Access

GET /api/v1/events (without Authorization header)

Expected: 401 Unauthenticated
{
  "message": "Unauthenticated."
}

Step 5 - Verify Admin Route Permission

GET /api/v1/admin/community-events using resident token

Expected: 403 Forbidden

2) Integration Workflow Screenshots

For actual defense/demo, these are the screenshots to capture in sequence:

  • Auth workflow: login request and returned token in Postman/Insomnia.
  • Stripe workflow: event/document request response containing checkout_url.
  • Email workflow: Mailtrap inbox receiving event notification email.
  • Notification workflow: API action then reflected in notifications endpoint/list.
  • QR workflow: successful registration token and scanner check-in response.

3) cURL Commands Used in Actual Testing

Login (Resident)

curl -X POST "https://your-domain.com/api/v1/auth/login" \
  -H "Accept: application/json" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "resident1@example.com",
    "password": "helloworld123",
    "device_name": "demo-client"
  }'

Use Token on Protected Endpoint

curl -X GET "https://your-domain.com/api/v1/auth/me" \
  -H "Accept: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN_HERE"

List Events (GET)

curl -X GET "https://your-domain.com/api/v1/events" \
  -H "Accept: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN_HERE"

Create Incident

curl -X POST "https://your-domain.com/api/v1/incidents" \
  -H "Accept: application/json" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN_HERE" \
  -d '{
    "category": "Noise Complaint",
    "description": "Loud music after curfew.",
    "incident_date": "2026-04-28",
    "respondent_email": "neighbor@example.com"
  }'

Negative Test (No Token)

curl -X GET "https://your-domain.com/api/v1/events" \
  -H "Accept: application/json"
